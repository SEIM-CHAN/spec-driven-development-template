# 実案件でのリポジトリ運用

このテンプレートは完成したアプリケーションではなく、実案件を始めるためのひな形です。案件ごとにこのテンプレートからリポジトリを作成し、要求・要件・設計・テスト・変更管理を継続して記録します。

## 推奨：コードと仕様書を同じリポジトリに置く

コードと仕様書が同じリリース単位で変わる案件では、次の構成を推奨します。

```text
customer-order-project/
├─ src/                    # 実装コード
├─ tests/                  # 自動テスト
├─ docs/                   # 要求・要件・設計・議事録・変更管理
├─ AGENTS.md               # Codexへの案件固有ルール
└─ README.md
```

### エンジニアが行うこと

- 要件・設計・コード・テストを同じ変更単位で更新する。
- プルリクエストに要求・要件ID、設計ID、テストIDを記載する。
- コードレビュー時に、仕様変更が必要なら `docs/` の更新も含める。

### エージェントに依頼すること

- 要件IDから関連する設計・コード・テストを検索し、影響範囲を提示する。
- プルリクエストの差分を要件と照合し、仕様外変更・設計漏れ・テスト漏れを指摘する。
- 実装後に追跡表のリンク切れを確認する。

### 向いているケース

- 一つのチームが仕様と実装を管理する。
- リリースと仕様変更のタイミングがほぼ同じである。
- 顧客資料をPrivateリポジトリに置ける。

## コードと仕様書を別リポジトリに置く場合

閲覧権限や管理単位が異なる場合は、仕様書リポジトリと実装リポジトリを分けます。

```text
specification-repo/        # このテンプレートを利用
application-repo/          # 実装コード
```

### エンジニアが行うこと

- Issue、ブランチ名、コミット、プルリクエスト、テスト結果に `FR-012` のような要件IDを記載する。
- 実装リポジトリのPR URL・コミットSHAを、仕様書リポジトリの `traceability.md` に記録する。
- 変更要求を承認してから、両方のリポジトリを更新する。

### エージェントに依頼すること

- 仕様書リポジトリと実装リポジトリの双方を読み、要件IDごとの対応関係を確認する。
- 変更要求から、更新が必要なリポジトリ・ファイル・テストを一覧化する。

### 向いているケース

- 顧客・業務部門と開発チームの閲覧権限を分けたい。
- 一つの仕様が複数のアプリケーションにまたがる。
- 既存のコードリポジトリの構成を変更できない。

## GitHubテンプレートから案件を開始する

1. GitHubでこのリポジトリを開き、`Use this template` を選ぶ。
2. 案件専用のPrivateリポジトリを作成する。
3. `PROJECT_CONTEXT.md` に案件名、承認者、対象範囲、リリース方針を記入する。
4. `docs/00_sources/` に受領資料の原本を保存し、一覧を更新する。
5. `docs/01_as_is/` で現行システムの実態を調査する。
6. 承認済みの要件から順に、設計・実装・テストへ進める。

既存コードがある場合は、テンプレートで作成した案件リポジトリへコードを移すか、既存コードリポジトリへ `docs/`、`AGENTS.md`、`PROJECT_CONTEXT.md` を取り込んでください。

## 要件からリリースまでのつなぎ方

```text
BR-001（業務要求）
  → FR-012（機能要件）
  → BD-014（基本設計）
  → DD-021（詳細設計）
  → 実装ブランチ / PR #123
  → AT-045（受入テスト）
```

このつながりを `docs/traceability.md` に記録します。仕様変更時は、変更要求 `CR-xxx` から影響を受けるIDをたどり、設計・コード・テストを更新します。

## 機密情報とアクセス権

- 顧客資料、個人情報、認証情報、接続先情報を含む場合は、案件リポジトリをPrivateにする。
- `docs/00_sources/` へ資料を入れる前に、契約・社内規程・顧客の持ち出しルールを確認する。
- パスワード、APIキー、秘密鍵、本番データはGitへコミットしない。
- Codexへ渡す資料も、利用を許可された範囲に限定する。
