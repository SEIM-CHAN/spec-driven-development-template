# 仕様統制ルール

## 情報源の優先順位

矛盾が起きた場合は、次の順で扱います。上位の情報を採用する前に、関連する要件・設計・テストへの反映要否を確認します。

1. 契約、法令、セキュリティ上の必須条件
2. `Approved` の意思決定記録・変更要求
3. `Approved` の要求・要件
4. `Approved` の基本・詳細設計
5. 受領原本・旧仕様書
6. 現行コード、テスト、運用実績
7. 口頭説明、推測、一般的慣習

現行コードは「現在の動作」の証拠であり、業務上の正しい仕様を保証するものではありません。不整合は `discrepancies.md` に記録し、勝手に解消しません。

## 状態と実装可否

| 状態 | 意味 | 実装可否 |
|---|---|---|
| Draft | 作成・調査中 | 不可 |
| Proposed | 承認待ち | 不可 |
| Approved | 承認済み | 可 |
| Rejected | 不採用 | 不可 |
| Deferred | 延期 | 不可 |
| Superseded | 別の仕様で置換済み | 不可 |

暫定実装が必要な場合は、承認者、期限、撤去条件、関連する仮定IDを明記します。

## 変更時の原則

変更要求 `CR-xxx` は、影響分析と承認が済むまで実装仕様ではありません。承認後は、要求・要件・設計・見積・テスト・追跡表を更新します。
