# ドキュメント運用ガイド

このディレクトリは、仕様を「根拠付きで決め、設計・実装・テストまで追跡する」ための正本です。エンジニアもエージェントも、未承認の内容や根拠のない推測を実装仕様として扱いません。

## フォルダごとの役割

| フォルダ | エンジニアが行うこと | エージェントに依頼すること | 残す成果物 |
|---|---|---|---|
| `00_sources/` | 顧客・既存案件から受領した原本を版数付きで保管する。原本は編集しない。 | 資料の目録化、要点抽出、相互矛盾・不足情報の検出。ただし原本は変更しない。 | 資料一覧、原本ファイル |
| `01_as_is/` | 現行コード、DB、画面、API、バッチ、運用を調査し、実態を確認する。 | コード調査、依存関係・処理フローの可視化、資料と実装の差分抽出。事実と推測を分離して報告する。 | 構成図、現行フロー、不一致一覧、リスク一覧 |
| `02_requests/` | 業務課題、目的、対象・対象外、優先度、成功指標を顧客と合意する。 | ヒアリング記録や資料から要求候補を整理し、曖昧な表現・確認事項を抽出する。 | `BR-xxx` 要求定義 |
| `03_requirements/` | 要求を検証可能な機能・非機能要件と受入条件へ分解し、承認を得る。 | 正常系・異常系・境界値・権限・監査・連携の不足をレビューする。承認前の要件は実装しない。 | `FR-xxx`、`NFR-xxx` などの要件定義 |
| `04_basic_design/` | 画面、API、外部連携、データ、権限、エラー、運用を外部仕様として設計する。 | 要件との漏れ・矛盾を照合し、利用者視点と既存機能への影響をレビューする。 | `BD-xxx` 基本設計 |
| `05_detailed_design/` | モジュール、処理順、状態遷移、DB、例外、トランザクション、排他を実装可能な粒度で決める。 | 既存コードとの整合、例外処理、並行実行、性能、テスト観点をレビューする。 | `DD-xxx` 詳細設計 |
| `06_tests/` | 要件に対応したテストケースを作成・実施し、実績と証跡を残す。 | 要件からテストケースを生成・レビューし、テスト漏れと回帰範囲を指摘する。 | `AT-xxx`、テスト結果、証跡 |
| `estimates/` | 作業分解、工数、単価根拠、リスク、予備費、変更差分を見積IDで管理し、承認を得る。 | 要件から作業を分解し、見積漏れ・前提・変更差分を整理する。ただし単価・生産性・最終金額は推測しない。 | `EST-xxx` 見積書、見積方針 |
| `hearings/` | 毎週の顧客ヒアリングで質問・回答・宿題を記録し、回答者を明記する。 | 前回差分と未解決質問から質問票を作成し、議事録から決定候補・質問・変更候補を抽出する。 | `H-xxx` 議事録 |
| `decisions/` | 誰がいつ何を正式決定したか、選択理由と影響を記録し、承認状態を管理する。 | 議事録から決定候補を分離し、関連する要件・設計・テストを特定する。 | `DEC-xxx` 意思決定記録 |
| `changes/` | 仕様変更の理由、変更前後、要件・設計・実装・テスト・納期への影響を承認前に評価する。 | 変更の影響範囲と回帰テスト対象を調査し、未更新ドキュメントを指摘する。 | `CR-xxx` 変更要求・影響分析 |

## 台帳ファイルの役割

| ファイル | エンジニアが行うこと | エージェントに依頼すること |
|---|---|---|
| `open_questions.md` | 実装判断を止める未確認事項をID付きで登録し、顧客へ確認する。 | 仕様・コード・ヒアリングから未解決事項を抽出し、影響度と質問案を提示する。 |
| `assumptions.md` | 暫定判断を根拠・期限・影響とともに記録し、確定前に見直す。 | 推測や補完を仮定として明示し、期限切れ・未確認の仮定を検出する。 |
| `glossary.md` | 業務用語、状態名、表記を統一する。 | 用語の揺れ、同義語、仕様間で異なる定義を検出する。 |
| `traceability.md` | 要求から要件、設計、実装、テストまでをIDで結び、変更時に更新する。 | リンク切れ、未テスト要件、根拠のない実装変更を検出する。 |
| `specification_governance.md` | 情報源の優先順位、承認状態、変更時の更新対象を守る。 | 情報の矛盾を検出し、承認前の内容を実装仕様として扱わない。 |
| `discrepancies.md` | 資料・コード・テスト・顧客回答の不整合を記録し、責任者と期限を設定する。 | 不整合候補を根拠付きで抽出し、勝手に解消しない。 |

## 実装の前後で守ること

### 実装前

- エンジニアは、対象要件が `Approved` であること、受入条件があること、未解決事項が実装を止めないことを確認する。
- エージェントは、影響範囲、変更予定ファイル、テスト計画、不明点を先に提示する。推測で仕様を補完しない。

### 実装後

- エンジニアは、設計・追跡表・テスト結果を更新し、レビューを受ける。
- エージェントは、要件との対応、変更ファイル、テスト結果、残課題・仮定を報告する。

## `.github/` の役割

プルリクエストのテンプレートを置く場所です。エンジニアは関連する要件IDと検証結果を記入し、エージェントはその情報を用いて仕様外変更、テスト漏れ、未解決の質問・仮定をレビューします。
